## 一、现状总览

虾饺 IM 已完成 6 个阶段的系统性改进（安全加固 → 通信可靠性 → 前端性能 → 体验优化 → 工程化 → PWA），在基础设施层面已达到较好水平。本文聚焦**尚未解决的深层问题**，评估产品方向与技术短板。

### 当前功能矩阵

| 层级 | 功能 | 完成度 |

## 二、核心问题诊断

### 2.1 Agent 能力体系薄弱（最关键差距）

虾饺 IM 的 Agent 目前只是**"会说话的机器人"**，而非**"会做事的 Agent"**。

| 维度 | 虾饺 IM 现状 | OpenClaw 做法 | 差距评级 |
|------|-------------|--------------|---------|
| **工具调用** | 零能力，只能纯文本对话 | 完整 function calling：web_search、memory_*、sessions_send | 🔴 致命 |
| **Agent 记忆** | 仅靠最近 20 条聊天历史 | memory/ 目录 + memory_write/search 工具，持久化记忆 | 🔴 严重 |
| **工作区协议** | 只读 SOUL.md | 7 文件协议：SOUL + IDENTITY + USER + TOOLS + AGENTS + HEARTBEAT + BOOTSTRAP | 🟡 中等 |
| **Agent 互调** | 仅群聊 @mention | agentToAgent RPC + sessions_send，Agent 可直接调用另一个 Agent | 🟡 中等 |
| **自主行为** | 无（被动应答） | HEARTBEAT.md + 定时 cron 任务，Agent 定期自主巡检、汇报 | 🟡 中等 |

**关键洞察**：没有 Tool Calling 的 Agent 只是聊天机器人。Tool Calling 是 Agent 从"能说"到"能做"的分水岭。

### 2.2 功能发散、不够垂直

当前功能覆盖了 IM + AI + 协作 + 知识 + 社交五个维度，但**每一层都做了，每一层都浅**：

| 功能 | "做了" | "没做深" |
|------|--------|---------|
| **RAG** | 有检索 | 无混合搜索、无重排、无质量评估 |
| **Workflow** | 有多步编排 | 无条件分支、无循环、无错误处理链 |
| **社区(虾区)** | 有帖子/评论 | 无内容推荐、无标签筛选、体验粗糙 |
| **定时例会** | 有 cron | 无会议纪要自动生成、无上下文记忆 |
| **图片生成** | 有 [IMG_GEN] | 无多轮修图、无风格选择 |

**产品定位建议**：虾饺 IM 的核心定位应该是 **"AI Agent 的协作操作系统"**，而非通用 IM 或社交平台。

- **深度强化**：Tool Calling + RAG + Workflow（核心差异化三件套）
- **保持现状**：IM 聊天、群组、消息管理（已经够用）
- **降级或精简**：社区(虾区)的 SNS 属性太重，可降级为"Agent 动态日志/工作看板"

### 2.3 技术基建待强化

| 领域 | 问题 | 影响 |
|------|------|------|
| **RAG 检索质量** | 演示级实现，生产使用必然不准 | 详见第三节专项分析 |
| **状态管理** | groups.json, workflows.json 等大量 JSON 文件，无事务保证 | 进程异常可能丢数据 |
| **前端架构** | 24 个 Vanilla JS 文件，全局变量交叉引用 | 维护成本递增 |
| **测试覆盖** | 零测试（无 test 目录、无测试框架） | 重构/迭代心智负担大 |
| **消息搜索** | 仅限当前频道，无全局搜索 | 用户找历史消息困难 |

## 三、RAG 知识库专项分析

### 3.1 当前实现扫描

虾饺 IM 的 RAG 位于 `server/services/rag.js`，核心架构：

```
文档 → 段落/句子分块(500字) → embedding(text-embedding-v3, 1024维)
  → 内存存储(.rag-index.json) → cosine 相似度暴力搜索 → top-5 返回
```

### 3.2 问题清单（对照业界标准）

| # | 问题 | 现状 | 业界标准 | 严重度 |
|---|------|------|---------|--------|
| 1 | **无混合检索** | 纯 embedding 向量搜索 | BM25 关键词 + 向量语义双路融合（RRF 加权） | 🔴 P0 |
| 2 | **无重排序** | cosine 相似度直出 top-K | 粗排召回 top-20 → Cross-Encoder / LLM 精排 | 🔴 P0 |
| 3 | **分块策略粗糙** | 按段落→句子，固定 500 字符 | 语义分块或分层分块（小块搜索 + 大块返回） | 🟡 P1 |
| 4 | **无元数据过滤** | 搜索遍历全部 chunk | 按文件名、类型、时间过滤缩小范围 | 🟡 P1 |
| 5 | **全内存 O(n) 搜索** | 所有 chunk embedding 加载内存，逐一比较 | 向量数据库（FAISS/hnswlib）ANN 索引 | 🟡 P1 |
| 6 | **无查询改写** | 用户原始查询直接 embedding | LLM 改写 / HyDE（假设答案搜索） | 🟠 P2 |
| 7 | **无上下文压缩** | 检索到的 chunk 全文拼接 | 摘要模型提取最相关段落，减少噪声 | 🟠 P2 |
| 8 | **无质量评估** | 无任何指标 | Recall@K、MRR、NDCG；用户反馈闭环 | 🟠 P2 |
| 9 | **索引非增量** | 文件变更重建整个文件 chunk | 按 chunk 粒度增量更新 | 🟠 P2 |
| 10 | **模型绑定硬编码** | 维度 1024 写死 | 索引元数据记录模型名+维度，自动检测不匹配 | 🟠 P2 |

### 3.3 典型失败场景

**场景 1：同义词盲区**
```
用户："我们的退款政策是什么？"
知识库："第七条 售后服务 7.1 退换货规则..."

→ "退款政策" 与 "退换货规则" 语义向量距离较远
→ 而 "费用结算和付款政策..." 反而向量更近
→ 返回错误的 chunk → LLM 给出错误回答
→ 如有 BM25："退"字关键词匹配会把正确文档拉上来
```

**场景 2：长文档上下文丢失**
```
用户："第三步怎么操作？"
知识库：一份操作手册，步骤分布在不同段落

→ "第三步" 被切分到一个 chunk 中，但缺少前后步骤上下文
→ LLM 拿到的是片段信息，无法完整回答
→ 如用分层分块：搜索小块定位 → 返回包含的大块保留上下文
```

**场景 3：技术术语精确匹配**
```
用户："如何配置 NGINX 的 proxy_pass？"
知识库：有专门的 NGINX 配置文档

→ 纯语义搜索可能返回 Apache 的代理配置（语义接近）
→ BM25 精确匹配 "NGINX" + "proxy_pass" 会直接命中
```

### 3.4 改进路线图

```
Phase 1（立竿见影，预计解决 ~40% 检索失败）：
  ├── 引入 BM25 关键词搜索
  │   → 可复用已有的 SQLite FTS5（messages_fts 同款技术）
  │   → 新建 chunks_fts 虚拟表存储 chunk 文本
  ├── RRF 融合排序
  │   → score = 1/(k+rank_bm25) + 1/(k+rank_vec)，k=60
  └── 阈值调优
      → 当前 0.3 固定阈值过低，应按查询长度动态调整

Phase 2（中期提升，Recall@5 提升 30-50%）：
  ├── LLM 重排序
  │   → 对 top-20 候选调用 LLM 打相关性分（0-10 分）
  │   → 选 top-5 高分 chunk 作为最终上下文
  ├── 分层分块
  │   → 搜索粒度：200 字小块（精准定位）
  │   → 返回粒度：包含小块的 800 字父块（保留上下文）
  ├── 元数据过滤
  │   → 搜索时可指定文件名/类型前缀过滤
  └── 查询改写
      → 用 LLM 将模糊查询改写为更明确的版本

Phase 3（长期完善，达到生产级）：
  ├── 引入 hnswlib-node（轻量 ANN 索引，替代暴力搜索）
  ├── 检索质量仪表盘（命中率、用户反馈标注）
  ├── HyDE 策略（先生成假设答案 → 用答案做检索）
  └── 索引增量更新（chunk 粒度 diff）
```

## 六、产品定位建议

### 当前定位（模糊）
> 虾饺 IM = IM + AI 对话 + 社区 + 知识库 + 工作流 + ...

### 建议定位（聚焦）
> **虾饺 IM = AI Agent 协作操作系统**
> 
> 核心价值：让多个 AI Agent 在统一平台上**接受任务、调用工具、协作完成、汇报结果**
> 
> 差异化三件套：**Tool Calling × RAG × Workflow**

### 功能取舍建议

| 功能 | 建议 | 原因 |
|------|------|------|
| IM 聊天/群组 | ✅ 保持 | 核心交互层，已足够好 |
| Tool Calling | ⬆️ 新增 | 最关键缺失能力 |
| RAG 知识库 | ⬆️ 深度强化 | 当前质量不足以生产使用 |
| Workflow | ⬆️ 增强 | 已有基础，需补齐分支/循环/错误处理 |
| Agent Memory | ⬆️ 新增 | 多轮对话必须 |
| 社区(虾区) | ⬇️ 精简 | 降级为"Agent 工作日志"即可 |
| 成长看板 | ⬇️ 精简 | 可合并到 Agent 详情页 |
| 定时例会 | ➡️ 保持 | 有用但优先级低 |

## 七、垂直方向：多 Agent 实时协作 IM

### 7.1 竞品格局与市场空白

| 竞品 | 定位 | 核心能力 | 用户群 | 局限 |
|------|------|---------|--------|------|
| **Coze/扣子** | Bot 构建平台 | 单 Agent + 插件市场 + 可视化 Workflow | 非技术用户 | 一次只和一个 Agent 对话 |
| **Dify** | LLMOps 平台 | 单 Agent + RAG + 工作流编排 | 开发者 | 无 IM 交互，HTTP 非实时 |
| **FastGPT** | 知识库问答 | RAG 深度优化 + 分享 | 企业客服 | 纯问答，无协作 |
| **CrewAI / AutoGen** | 多 Agent 框架 | 代码级多 Agent 编排 | 纯开发者 | 无 UI，非终端用户产品 |
| **Slack + AI Bots** | 协作 IM | IM + 零散 Bot 插件 | 企业团队 | Bot 之间互不协作 |
| **ChatGPT / Claude** | 通用对话 | 单轮/多轮对话 | 所有人 | 单 Agent，无团队概念 |

**市场空白**：

- **单 Agent 工具** 已经饱和（Coze、Dify、FastGPT 等）
- **多 Agent 框架** 只面向开发者（CrewAI、AutoGen、MetaGPT），没有终端用户产品
- **IM 产品** 没有把 AI Agent 当作"团队成员"来设计

**结论**：没有人在做一个**终端用户可用的、多 Agent 实时协作的 IM 产品**。这是虾饺 IM 的最佳切入点。

### 7.2 核心定位

> **虾饺 IM = AI 团队的 Slack**
>
> 你不是在和一个 AI 聊天，你是在**指挥一个 AI 团队**完成任务。

#### 与竞品的本质差异

| 维度 | 单 Agent 产品（Coze/Dify/ChatGPT） | 虾饺 IM（多 Agent 协作） |
|------|----------------------------------|----------------------|
| **用户心智** | "我在用一个工具" | "我有一个团队" |
| **能力上限** | 单个模型的极限 | N 个专家组合无上限 |
| **交互方式** | 一问一答 | 群聊讨论、任务接力 |
| **知识边界** | 一个知识库 | 每个 Agent 自己的知识库 + 共享知识库 |
| **用户粘性** | 换一个更好的 Bot 就走 | 整个团队配置好了舍不得走 |
| **使用场景** | 单点任务（翻译、写文案） | 复合任务（调研 → 分析 → 执行 → 汇报） |

### 7.3 核心使用场景

#### 场景 A：产品调研团队

```
用户在一个群聊中组建了 3 个 Agent：市场分析师、竞品专家、文案策划

用户：帮我调研宠物智能喂食器市场

→ 市场分析师（自动响应，因为与"市场"相关）
  → [Tool Call: web_search("宠物智能喂食器 市场规模 2026")]
  → [Tool Call: web_search("宠物经济 增长趋势")]
  → 输出：市场规模报告

→ 竞品专家（收到市场分析师的输出后自动接力）
  → [Tool Call: web_search("小佩 PETKIT 智能喂食器 评测")]
  → [Tool Call: rag_query("竞品分析模板")]
  → 输出：竞品对比表 + 差异化建议

→ 文案策划（基于竞品专家的差异化建议）
  → 输出：3 版营销文案 + 卖点提炼

用户全程看着他们协作，随时插话调整方向
```

#### 场景 B：代码审查团队

```
群组成员：架构师 Agent、安全审计 Agent、性能优化 Agent

用户：审查这段代码（粘贴代码）

→ 架构师：从设计模式、可维护性角度点评
→ 安全审计：从 XSS、SQL 注入、权限漏洞角度扫描
→ 性能优化：从复杂度、内存、并发角度分析

三个 Agent 各自输出后，架构师自动汇总为一份综合审查报告
```

#### 场景 C：内容创作流水线

```
群组成员：选题策划 Agent、写作 Agent、配图 Agent

用户：写一篇关于 AI 编程的公众号文章

→ 选题策划：分析热点，输出 3 个选题 + 大纲
→ 用户选择其中一个
→ 写作 Agent：基于大纲 + [Tool Call: rag_query] 查知识库，产出全文
→ 配图 Agent：基于段落关键词，[Tool Call: image_gen] 生成配图
→ 最终输出：图文并茂的完整文章
```

### 7.4 要做到出彩的 3 个关键能力

#### 关键能力 1：Agent 间的主动协作

**现状**（人工驱动）：
```
人类 @Agent_A "分析市场" → Agent_A 回复
人类 @Agent_B "基于上面的分析做竞品对比" → Agent_B 回复
人类 @Agent_C "基于竞品对比写文案" → Agent_C 回复
```

**目标**（自动接力）：
```
人类 "帮我调研宠物喂食器市场"
→ Agent_A 自动分析市场，完成后 @Agent_B 传递结果
→ Agent_B 自动做竞品对比，完成后 @Agent_C 传递结果
→ Agent_C 自动写文案，完成后汇报给人类
```

**技术实现**：

| 组件 | 说明 |
|------|------|
| **call_agent 工具** | Agent A 通过 Tool Call 调用 Agent B，传递任务描述和上下文 |
| **协作意图识别** | 分析用户消息中涉及的能力域，自动路由到对应 Agent |
| **任务链编排** | 在群组设置中定义 Agent 间的协作链：A → B → C |
| **上下文传递** | Agent B 接收 Agent A 的输出作为输入上下文 |
| **人类控制点** | 每个接力节点可设置"需要人类确认"或"自动继续" |

#### 关键能力 2：Tool Calling（让 Agent 做事）

**内置工具集（按优先级）**：

| 工具 | 功能 | 优先级 |
|------|------|--------|
| `rag_query` | 查询当前 Agent 的知识库 | P0 |
| `web_search` | 搜索互联网获取实时信息 | P0 |
| `call_agent` | 调用另一个 Agent 完成子任务 | P0 |
| `memory_write` | 将关键信息写入 Agent 持久记忆 | P0 |
| `memory_search` | 搜索 Agent 历史记忆 | P0 |
| `image_gen` | 生成图片（替代当前 [IMG_GEN] 标记） | P1 |
| `code_run` | 在沙箱中执行代码片段 | P1 |
| `file_read` | 读取工作区文件 | P1 |
| `file_write` | 写入工作区文件 | P1 |
| `http_request` | 调用外部 API | P2 |

**技术实现**：

```
用户消息 → LLM（带 tools 参数）
  → LLM 返回 tool_calls
  → 服务端执行工具
  → 结果注入上下文
  → LLM 继续生成（可能再次 tool_call）
  → 循环直到 LLM 输出最终文本
  → 流式推送给前端
```

前端展示增强：
- 工具调用过程实时展示（"正在搜索..."、"正在查询知识库..."）
- 工具结果可折叠查看（用户可以检查 Agent 查到了什么）
- 失败的工具调用显示错误原因和重试按钮
