# P3.4 SQLite 改造 — 安全与质量审查报告

**审查日期**: 2025-03-14  
**审查范围**: database.js, storage.js, messages.js, community.js, agents.js, chat.js

---

## 1. SQL 注入

### 结论：✅ 通过

| 文件 | 检查项 | 结果 |
|------|--------|------|
| database.js | 迁移脚本、建表语句 | 全部使用 `prepare().run/all()` 参数化 |
| storage.js | 所有 CRUD 操作 | 使用 `?` 占位符，无字符串拼接 |
| storage.js | `bumpMetric` 的 `${key}` | `key` 经 `validKeys` 白名单校验，安全 |
| storage.js | `searchMessages` FTS 查询 | `ftsQuery` 作为参数传入 `MATCH ?`，非拼接 |
| messages.js | channel / before / limit | 均通过 params 传递 |

**说明**：`searchMessages` 中对 keyword 的 `replace(/"/g, '""')` 为 FTS5 短语搜索转义，最终仍通过参数绑定，无 SQL 注入风险。

## 3. 权限控制

### 结论：⚠️ 需改进

| API | 当前状态 | 建议 |
|-----|----------|------|
| `GET /api/messages` | 无 `guardOwner` | 建议增加 `isOwnerReq` 校验，避免未登录读取消息 |
| `GET /api/messages/search` | 无 `guardOwner` | 同上 |
| `DELETE /api/community/posts/:id` | 有 `guardOwner` ✅ | - |
| `DELETE /api/agents/:id` (cascade) | 有 `guardOwner` ✅ | - |
| `GET /api/agents/:id/delete-impact` | 有 `guardOwner` ✅ | - |

**说明**：messages 为只读接口，前端通过 `authFetch` 带 token，但服务端未校验。若为内网/单用户部署可接受；若需多用户隔离，应增加 `isOwnerReq(req)` 校验。

## 4. 错误处理

### 结论：⚠️ 部分缺失

| 位置 | 状态 | 说明 |
|------|------|------|
| `searchMessages` | ✅ | try/catch 返回 `{ results: [], total: 0 }` |
| `addMessage` | ❌ | 无 try/catch，DB 异常会向上抛出 |
| `queryMessages` | ❌ | 无 try/catch |
| `addPost` / `addComment` | ❌ | 无 try/catch |
| `deletePost` / `deletePostsByAuthor` | ❌ | 无 try/catch |
| `toggleReaction` | ✅ | 有 try/catch + ROLLBACK |
| `bumpMetric` / `addRating` | ✅ | 有 try/catch |
| messages.js handle | ⚠️ | 异常由 router 统一捕获返回 500 |

**建议**：对关键写操作（addMessage、addPost 等）增加 try/catch，失败时记录日志并返回明确错误，避免未处理异常导致 500。

## 6. DoS 防护

### 结论：⚠️ 需改进

| 检查项 | 当前 | 建议 |
|--------|------|------|
| `/api/messages` limit | `Math.min(parseInt(...), 100)` | 增加 NaN 兜底：`parseInt(...) \|\| 30` |
| `/api/messages/search` limit | `Math.min(parseInt(...), 50)` | 同上：`parseInt(...) \|\| 20` |
| search API 频率限制 | 无 | 可考虑简单 rate limit（如每 IP 60 次/分钟） |
| `getAllPosts` | 无 limit | 见下方「性能」 |

**NaN 问题**：`parseInt('x', 10)` 为 NaN，`Math.min(NaN, 100)` 仍为 NaN，传入 SQLite `LIMIT` 可能异常。建议：

```javascript
const limit = Math.min(parseInt(query.get('limit') || '30', 10) || 30, 100);
```

## 8. 性能

### 结论：⚠️ 存在风险

| 检查项 | 问题 | 建议 |
|--------|------|------|
| `_postRow` N+1 | 每个 post 调用 `getComments`、`getReactions` | 可接受（帖子数通常有限），大规模时考虑批量查询 |
| `getAllPosts` | 无 limit，加载全部 posts | 高风险：帖子多时可能 OOM |
| community 路由 | `getAllPosts().slice(0, 100)` | 先取全量再截断，浪费内存和 CPU |

**建议**：在 storage 层为 `getAllPosts` 增加 limit 参数，或新增 `getRecentPosts(limit)`，community 路由改为调用带 limit 的接口。

---

## 问题汇总

### 高优先级

1. **messages API 无鉴权**：`GET /api/messages`、`GET /api/messages/search` 未校验 `isOwnerReq`，未登录可读消息。
2. **limit 参数 NaN**：`parseInt` 失败时 limit 为 NaN，可能影响 SQL 执行，需加兜底。
3. **getAllPosts 无 limit**：帖子量大时存在 OOM 风险，且 community 仅用前 100 条。

### 中优先级

4. **写操作错误处理**：`addMessage`、`addPost`、`addComment` 等无 try/catch，DB 异常会直接抛出。
5. **search API 无 rate limit**：可被高频请求消耗资源。

### 低优先级

6. **community 路由效率**：`getAllPosts().slice(0, 100)` 应改为带 limit 的查询。

---

## 审查结论

- **SQL 注入**：✅ 无问题  
- **XSS**：✅ 无问题  
- **权限**：⚠️ messages 接口建议加鉴权  
- **错误处理**：⚠️ 部分写操作缺少降级/日志  
- **内存**：✅ 无泄漏  
- **DoS**：⚠️ limit NaN 需修复，search 可加 rate limit  
- **数据一致性**：✅ 无问题  
- **性能**：⚠️ getAllPosts 需限制数量  

整体改造质量良好，建议优先处理 limit NaN、getAllPosts limit 以及 messages 鉴权三项。
