# 虾饺拆分 — 脱离 OpenClaw 独立运行技术方案

> 目标：让虾饺 IM 完全独立于 OpenClaw Gateway 运行，仅需一个 LLM API Key 即可启动。

---

## 一、当前依赖关系

### 1.1 Gateway 依赖（AI 能力层）

| 依赖点 | 代码位置 | 说明 |
|--------|----------|------|
| `sendToGateway()` | server.js L317 | 所有用户消息通过 Gateway 的 `chat.send` 转发给 Agent |
| Agent 流式响应 | server.js `handleGatewayMessage()` | Agent 回复（stream/lifecycle/chat）全部从 Gateway 回传 |
| `applyConfig()` | server.js L839 | 创建/修改/删除 Agent 时通过 `config.apply` 同步到 Gateway |
| 健康检查 | Gateway `health` 事件 | Gateway 提供在线 Agent 列表（`knownAgents`） |
| Gateway 认证 | server.js L229 | 使用 `GATEWAY_TOKEN` 通过 WebSocket 握手认证 |

**影响：Gateway 不可用 → Agent 完全无法回复消息**

### 1.2 配置文件依赖（Agent 管理层）

| 依赖点 | 路径 | 说明 |
|--------|------|------|
| `openclaw.json` | `%USERPROFILE%\.openclaw\openclaw.json` | Agent 列表、模型配置、工具配置的唯一来源 |
| Agent 工作区 | `%USERPROFILE%\.openclaw\workspace-{id}\` | SOUL.md、知识库文件的存储位置 |
| 模型列表 | 从 `openclaw.json` 的 `models` 字段读取 | 前端模型选择器的数据源 |
| Agent CRUD | 读写 `openclaw.json` + `config.apply` | 创建/修改/删除 Agent 都要同时操作配置文件和 Gateway |

**影响：`openclaw.json` 不存在 → 无法列出/创建/管理 Agent**

---

## 二、目标架构

```
当前：
  浏览器 → 虾饺 IM Server → OpenClaw Gateway → LLM API
                ↕
          openclaw.json（配置）
          workspace-{id}/（工作区）

目标：
  浏览器 → 虾饺 IM Server → LLM API（直连）
                ↕
          agents.json（自管理）
          data/workspace-{id}/（项目内）
```

## 三、拆分步骤

### Phase 1：解耦配置存储

**目标**：Agent 元数据和工作区不再依赖 `openclaw.json`

#### 1.1 自建 agents.json

- 在项目根目录创建 `agents.json`，格式：

```json
{
  "agents": [
    {
      "id": "wechat-editor",
      "name": "公众号选题",
      "emoji": "📋",
      "model": "deepseek/deepseek-chat",
      "workspace": "data/workspace-wechat-editor",
      "createdAt": "2026-03-01T00:00:00Z"
    }
  ]
}
```

- 迁移 `openclaw.json` 中的 agent 数据到 `agents.json`
- 提供迁移脚本 `migrate-from-openclaw.js`

#### 1.2 工作区迁移

- 将 `%USERPROFILE%\.openclaw\workspace-{id}\` 迁移到项目内 `data/workspace-{id}/`
- `resolveWorkspace()` 改为读取 `agents.json` 中的 `workspace` 字段
- SOUL.md、知识库文件位置不变，只是根目录变了

#### 1.3 模型配置独立

- 新建 `models.json` 或在 `agents.json` 中增加 `models` 字段：

```json
{
  "models": [
    { "id": "deepseek/deepseek-chat", "name": "DeepSeek Chat", "provider": "deepseek" },
    { "id": "openai/gpt-4o", "name": "GPT-4o", "provider": "openai" },
    { "id": "anthropic/claude-sonnet", "name": "Claude Sonnet", "provider": "anthropic" },
    { "id": "ollama/llama3", "name": "Llama 3 (本地)", "provider": "ollama" }
  ],
  "providers": {
    "openai": { "baseUrl": "https://api.openai.com/v1", "apiKey": "env:OPENAI_API_KEY" },
    "deepseek": { "baseUrl": "https://api.deepseek.com/v1", "apiKey": "env:DEEPSEEK_API_KEY" },
    "anthropic": { "baseUrl": "https://api.anthropic.com/v1", "apiKey": "env:ANTHROPIC_API_KEY" },
    "ollama": { "baseUrl": "http://localhost:11434/v1", "apiKey": "" }
  }
}
```

#### 涉及文件修改

| 文件 | 改动 |
|------|------|
| server.js | `OPENCLAW_CONFIG` 相关全部替换为读写 `agents.json` |
| server.js | `resolveWorkspace()` 改为基于 `agents.json` |
| server.js | `getModels()` 改为读 `models.json` |
| server.js | 删除 `applyConfig()` 及所有 `config.apply` 调用 |

---

### Phase 2：直连 LLM API

**目标**：不经过 Gateway，直接调用 LLM HTTP API

#### 2.1 新建 llm-client.js

统一的 LLM 调用模块，支持 OpenAI 兼容格式：

```javascript
// llm-client.js 核心接口
class LLMClient {
  constructor(config) { /* 读取 models.json / providers */ }

  // 非流式调用
  async chat(modelId, messages, options) → { content, usage }

  // 流式调用
  async chatStream(modelId, messages, options) → AsyncGenerator<{ delta, done }>
}
```

**支持的 Provider：**
- OpenAI / Azure OpenAI（`/v1/chat/completions`）
- DeepSeek（兼容 OpenAI 格式）
- Anthropic Claude（`/v1/messages`，需适配）
- Ollama（本地，兼容 OpenAI 格式）
- 其他 OpenAI 兼容 API（硅基流动、月之暗面等）

#### 2.2 替换 sendToGateway

| 原函数 | 替换为 | 说明 |
|--------|--------|------|
| `sendToGateway(channel, agentId, text)` | `sendToLLM(channel, agentId, text)` | 直接调 LLM API |
| Gateway `agent` 事件处理 | `llm-client.js` 流式回调 | 流式输出转 WebSocket 推送 |
| `gwSendReq(method, params)` | 删除 | 不再需要 Gateway RPC |
| `gwConnect()` | 删除 | 不再需要 Gateway WebSocket |

#### 2.3 消息构建

每次调用 LLM 时构建 messages 数组：

```javascript
async function sendToLLM(channel, agentId, userText) {
  const agent = getAgent(agentId);
  const soul = readSoulMd(agentId);     // system prompt
  const kbContext = loadKbContext(agentId); // 知识库上下文
  const recentMsgs = getRecentMessages(channel, 20); // 历史消息

  const messages = [
    { role: 'system', content: soul + '\n\n' + kbContext },
    ...recentMsgs.map(m => ({
      role: m.type === 'agent' ? 'assistant' : 'user',
      content: m.text
    })),
    { role: 'user', content: userText }
  ];

  const stream = llmClient.chatStream(agent.model, messages);
  // 流式推送到 WebSocket...
}
```

#### 涉及文件

| 文件 | 改动 |
|------|------|
| 新建 `llm-client.js` | LLM API 统一调用模块 |
| server.js | 删除 Gateway 相关代码（~200 行） |
| server.js | `sendToGateway` → `sendToLLM` |
| server.js | 流式响应改为 LLM stream → WebSocket broadcast |
| scheduler.js | `sendToGateway` → `sendToLLM` |

### Phase 3：兼容模式（可选）

**目标**：通过环境变量切换 standalone / gateway 模式

```bash
# 独立模式（默认）
MODE=standalone node server.js

# Gateway 模式（兼容 OpenClaw）
MODE=gateway GATEWAY_WS=ws://127.0.0.1:18789 node server.js
```

```javascript
// server.js 启动时
const MODE = process.env.MODE || 'standalone';

if (MODE === 'gateway') {
  // 现有 Gateway 逻辑
  gwConnect();
} else {
  // 直连 LLM
  const llmClient = new LLMClient(loadModelsConfig());
}
```

## 七、风险与注意事项

1. **工具调用（Function Calling）**：OpenClaw Gateway 支持 Agent 使用工具（如 agent-to-agent 通信）。独立模式初期可不支持，后续按需添加。

2. **多轮对话上下文**：Gateway 自带会话管理。独立模式需自行维护上下文窗口，注意 token 上限控制。

3. **并发控制**：多个用户同时对话时，需控制 LLM API 并发请求数，避免触发限流。

4. **费用控制**：直连 LLM API 时应记录 token 用量，可在 `agent-metrics.json` 中增加用量统计。

5. **知识库 RAG**：当前知识库文件直接拼接到 system prompt。文件多/大时可能超出上下文窗口，后续可引入向量检索（如 `@xenova/transformers` 本地 embedding）。

6. **向后兼容**：已有的 JSON 数据文件（`chat-history.json`、`groups.json` 等）格式不变，无需迁移。
